Checked boxes indicate bugs, style violations, and other problems detected during code review. Most of them can be easily located by searching through the patch file for the corresponding keywords or expressions.

Constructors that may be used with a single parameter should be explicit unless silent type conversion is desired. If silent type conversion is desired, non-explicit constructor declaration should have a comment about the conversion.
All non-static class data members must be initialized [in all class constructors]. You may skip data members that have reasonable default constructors.
Make pointers nil after delete unless they clearly go out of scope right after delete. For example, pointer deletions in small and simple destructors are excepted.
Do not check pointers for nil before deleting them. C++ does that for you.
Use (foo) and (!foo) instead of (foo != 0) and (foo == 0) when testing pointers for being nil.
Use (foo) and (!foo) instead of (foo == true), (foo == false), and such when testing boolean expressions for being true/false.
Avoid multiple virtual methods with identical names. Correctly overriding them is error-prone.
Use unified patch with at least 20 lines of context (-U 20).
Use --show-c-function or equivalent where supported.
A patch should not have whitespace-only changes unless they are intentional and in patch scope.
A patch should not have changes that only add or remove empty lines.
Use camelCaseNames when the code around your changes uses such names. Avoid adding names_with_underscores in that case.
Use names_with_underscores when the code around your changes uses such names. Avoid camelCase names in that case.
Do not prefix variable and data member names with their types (a.k.a. Hungarian notation altnough the original approach under that name was reportedly misunderstood and misused). Anyway, just name the thing; its type may change and is rarely critical for its purpose.
Avoid adding {} for single-line statements like 
                if (foo) {
                    bar();
                }
Order class sections as such: public, protected, and private.
Within each public/protected/private section, declare methods before data members.
Within each public/protected/private section, declare static members before non-static ones.
Use const when you can.
Declare local variables as close to their first use, including inside if-statement conditions when possible: if (const char *foo = ...).

This is an experimental interface. The interface itself and even the issue IDs may change.
